## 设计理念
XEngine 的目标不是“做一堆工具类”，而是提供一套可复用、可扩展、可部署的工程化基础设施，面向 **服务端/工具链/引擎型项目** 的长期演进。

### 1) 模块化（Modular）

- **职责分离**：网络、HTTP、WebSocket、定时器、任务系统、DB/Redis、压缩等能力以独立模块存在，避免“大一统单体”互相耦合。
- **可组合**：上层业务按需选择模块组装，不强绑定全家桶。
- **生命周期一致**：模块遵循统一的初始化/启动/更新/停止/销毁等生命周期，方便托管与自动化运维。

适用场景：网关、登录服、游戏服、微服务、后台工具、长连接服务等。

---

### 2) 插件化（Pluggable / Replaceable）

- **接口驱动**：引擎以接口定义能力边界（Api / Service / Module），实现可替换。
- **实现可替换**：同一接口可以有不同实现（例如不同网络后端、不同存储后端、不同平台实现），上层业务代码不变。
- **按需加载**：插件/模块可以按配置启用或禁用，支持灰度启用新能力。

这让 XEngine 具备“长期迭代”的工程属性：你可以在不破坏业务代码的前提下，替换底层实现或新增能力。

---

### 3) 高并发（High Concurrency）

XEngine 面向高吞吐服务，设计上强调：

- **事件驱动 / 异步化**：尽量避免阻塞式调用，降低线程数与上下文切换成本。
- **任务系统解耦**：将 IO、计算、定时任务分离，避免在网络线程里做重活。
- **资源复用**：连接、缓冲、对象、线程等尽量池化/复用，减少频繁分配带来的抖动。
- **可观测性**：关键路径可统计（延迟、吞吐、队列堆积），便于压测与定位瓶颈。

> 备注：具体线程模型（例如单 Reactor / 多 Reactor、IO 线程与工作线程的划分）以工程配置与模块实现为准。

---

### 4) 跨平台（Cross-Platform）

XEngine 以 **CMake + 平台抽象层** 为核心策略，目标覆盖：

- Windows / Linux（优先）
- 其他平台可通过抽象层扩展（例如 macOS）

跨平台策略：

- **统一入口**：尽量避免平台相关代码散落在业务层，集中在平台抽象层中。
- **统一构建**：CMake 管理依赖、编译选项、输出目录与安装规则。
- **最小平台差异**：封装文件系统、线程、时间、socket 等差异点。

---

## 编译方式（CMake）

### 1) 环境要求

- **CMake**：建议 3.20+（更推荐较新版本）
- **编译器**：
  - Windows：Visual Studio 2022（MSVC x64）
  - Linux：GCC / Clang（建议 GCC 11+ 或 Clang 14+）
- （可选）依赖库：按模块启用情况决定（例如 protobuf、openssl、zlib、mysql client、hiredis 等）

---

### 2) Windows（Visual Studio / MSVC）

推荐 out-of-source 构建：
